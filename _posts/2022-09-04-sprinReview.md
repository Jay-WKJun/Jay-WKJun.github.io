# 배운 것들

개인적으로 이번 협업 경험을 하면서 배운 것들이 정말 많습니다.

누구나 머리로는 아는 내용이고, 저 또한 익히 들어 알고 있는 사실이지만, 직접 경험해보니 정말 피부로 와닿는 느낌이었습니다. (절대 잊지 못할 것 같습니다.)

최근까지 코로나로 제대로 된 협업 기회를 얻기 쉽지 않았는데, 정말 행운이었다고 생각합니다.

## 구체적이고 명확히 볼 수 있어야 이해할 수 있다.

레퍼런스를 모으고 구체적인 와이어프레임을 짜는데 있었던 내 얘기

툴을 잘 못써서 구체적인 시각자료를 잘 모으지 못함

주변 서비스들을 모아 짜집기해서 시각자료를 보충하고 말로 설명

엉망인 내 사진

직접 만든 와이어프레임으로 자신의 아이디어와 의견을 전달한 사람들이 훨씬 더 많은 호응과 이해를 받았다.

협업에서 시각적 자료의 힘을 배움

## 책임지지 않는 의견은 의미가 없다.

누구하나 강제성 없는 프로젝트에서 어떤 분야에 특정 책임자 없이 진행하는 것은 평행선을 달릴 뿐이다.

초반에 분류를 위한 데이터 구조와 UI 디자인을 선정할 당시,

멀티 태그 형식 vs 전통적인 directory 형식의 카테고리 의견이 부딪침.

책임자가 없으니, 의견이 통합되지 않았음.

테오의 개입으로 책임자를 정하고 그 사람의 주도하에 작품을 이어감.

이전까지는 모두 평등한 상태로 치열한 토론을 통해 가장 최선을 고르는 것이 최고라고 생각했다.

하지만, 특정 분야에 뛰어난 사람을 믿어주는 것도 성공적인 프로젝트를 위해 필요하다고 생각함.

그것이 책임을 가진 사람의 존중감을 높여주어 퍼포먼스를 높이고 커뮤니케이션 비용을 줄이며 팀 전체의 결속력을 가져올 수 있기 때문이라고 생각했다.

실제로 책임자가 정해지고 모두가 책임자를 믿어주자, 결정도 일사분란하게 되었고, 두가지 의견을 절충한 더 훌륭한 결과물이 나왔기 때문이다.

## 개발 스타일은 지구의 인구 수 만큼 천차만별이다, lint와 협업 규칙의 중요성

팀원 중엔 현업에서 일하는 사람도, git을 처음 써보는 사람도 있었다.

코딩을 시작하기에 앞서 구체적인 브랜치 전략을 설명하고 협업 방식을 설명했지만, 그 와중에도 엄청난 conflict가 발생했다...

따라서, 8월 4일(토) 새벽을 꼬박 세워 린트가 적용돼있지 않던 프로젝트에 lint + prettier를 적용해 코드 스타일을 맞추었다.

적용된 lint 사진

거기에 PR description에 익숙하지 않은 팀원을 위해 PR template을 만들어 고민하지 않고 자세한 description을 쓸 수 있도록 했다.

PR template 사진

### 코드의 일관성을 지킨다는 것.

코드의 일관성은 가독성 뿐만아니라 merge와 review에도 크게 도움 준다는 사실을 알았다.

lint를 적용하고 코드 스타일이 통일 되어 코드 review가 훨씬 편해졌고,

Lint를 적용하기 전에 분명 같은 코드인데 코드 스타일 변화로 인해 conflict가 발생했었는데, 그런 일이 확연하게 줄었다.

conflict 사진

### 중요한 협업 규칙은 시스템으로 강제되어야 한다.

협업 규칙을 1시간에 걸쳐 설명했지만, git에 익숙하지 않은 사람들은 결국 실수를 하기 마련이었다.

대표적으로 main 브랜치를 force push해서 그동안 작업을 모두 뒤엎는 일도 있었다.

이는, 그동안의 사람들의 수고를 한방에 날리는 행위로,,, 악의는 없었다고 하지만, 치명적이었다.

물론 마침 동기화 된 dev 브랜치가 있어 금방 복구할 수 있었다.

그동안 Lint와 branch 규칙의 강제성의 중요성을 머리로만 알고 있었다면, 이번에 피부로 느끼게 되었다.
